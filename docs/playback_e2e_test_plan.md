# 播放链路全链路测试方案

## 目标

验证本地播放链路在真机上的关键能力：

1. 首次播放可正常启动并进入稳定播放状态。
2. 二次播放可命中音频缓存，走本地文件播放。
3. 播放列表切歌、上一首/下一首、进度拖动状态一致。
4. 异常链路可恢复，包括缓存损坏、网络中断、歌曲信息缺失。
5. UI 状态、通知栏状态、内部播放器状态保持一致。

## 测试环境

- 设备：Android 真机 `MEIZU 21`
- 构建模式：`flutter run --debug`
- 网络：同一局域网，确保手机可访问当前服务端
- 日志关注点：
  - `createCachedAudioSource`
  - `AudioPlaybackCache`
  - `MyAudioHandler`
  - `播放器发生错误`
  - `已成功重建当前音源并恢复播放`

## 前置准备

1. 清空 App 进程，避免旧播放状态干扰。
2. 首轮测试前清理应用缓存，确保“首次播放”是冷启动路径。
3. 确认可访问歌曲接口与歌单接口。
4. 准备三类测试歌曲：
   - 常规可播放歌曲 2-3 首
   - 首次无缓存歌曲 1 首
   - 元数据不完整或容易失败的边界歌曲 1 首

## 用例清单

### 1. 冷启动单曲播放

步骤：

1. 启动 App。
2. 选择一首确认可播放的歌曲。
3. 点击播放。

预期：

- 进入播放页后 3 秒内开始出声。
- UI 显示 `isPlaying=true`，进度持续推进。
- 通知栏显示当前歌曲信息。
- 日志应出现远程 URL 音源创建，不应出现实验性 API 告警。

### 2. 二次播放缓存命中

步骤：

1. 完成用例 1 后暂停并返回。
2. 再次播放同一首歌。

预期：

- 启动速度快于首次播放。
- 日志可看到命中本地缓存文件并创建 `AudioSource.file`。
- 无明显首帧卡顿。

### 3. 播放列表连续切歌

步骤：

1. 从歌单页播放包含至少 3 首歌的歌单。
2. 执行下一首、上一首、点击队列第 N 首。

预期：

- 当前歌曲标题、索引、播放按钮状态一致。
- 不出现“点 A 播 B”。
- 最后一首切下一首时行为符合当前循环模式。

### 4. 进度拖动

步骤：

1. 播放一首时长较长的歌曲。
2. 拖动进度条到中间和接近结尾。

预期：

- 播放位置快速跳转。
- UI 进度与实际音频位置同步。
- 不出现拖动后瞬间回跳。

### 5. 后台/前台切换

步骤：

1. 开始播放。
2. 切到后台 10 秒后返回。
3. 锁屏后再解锁。

预期：

- 播放不中断或按系统策略暂停，但 UI 与真实状态一致。
- 返回前台后当前歌曲、时长、进度不乱。

### 6. 缓存损坏恢复

步骤：

1. 先让歌曲完成一次缓存。
2. 手工破坏对应缓存文件或清理一半文件内容。
3. 再次播放该歌曲。

预期：

- 首次读本地文件失败后，系统删除坏文件并自动重建音源。
- 可回退到网络播放并恢复出声。
- 日志可看到“删除缓存并重建音源”。

### 7. 网络中断恢复

步骤：

1. 播放未缓存歌曲。
2. 在播放中途关闭 Wi-Fi/移动网络。

预期：

- 出现可预期错误日志。
- UI 不应永久停留在“正在播放”假状态。
- 若有下一首且策略允许，可自动跳过。

### 8. 缺失歌曲信息

步骤：

1. 播放一个本地缓存中无歌曲信息的歌曲。

预期：

- 会先尝试从接口补齐歌曲信息。
- 补齐成功后进入正常播放。
- 失败时 UI 回退到非播放态。

## 验收标准

- `flutter analyze` 通过。
- 真机启动成功。
- 8 个用例中，前 5 个必须通过。
- 后 3 个异常用例至少满足“不崩溃、不假播放、可恢复或可感知失败”。

## 建议执行顺序

1. 冷启动单曲播放
2. 二次播放缓存命中
3. 播放列表连续切歌
4. 进度拖动
5. 后台/前台切换
6. 缓存损坏恢复
7. 网络中断恢复
8. 缺失歌曲信息
