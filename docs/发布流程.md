# 发布流程文档

本文档描述了如何为项目打 tag、打包、修复版本号并发布到 GitHub 的完整流程。

## 📋 前置准备

在开始发布流程之前，请确保：

1. ✅ 所有代码已提交到本地仓库
2. ✅ 代码已推送到远程仓库（`git push`）
3. ✅ 已通过所有测试和代码检查
4. ✅ 已更新 CHANGELOG.md（如适用）
5. ✅ 已更新 `docs/RELEASE_NOTES.md` 作为本次 GitHub Release 正文

## 📝 Release Notes 规范

本项目只维护一份 Release Notes 文件：`docs/RELEASE_NOTES.md`。

发布时请遵循以下约定：

1. 每次正式发版前，直接更新 `docs/RELEASE_NOTES.md`
2. GitHub Actions 创建正式 Release 时，会直接读取这一个文件作为正文
3. 不再为每个版本额外创建 `docs/RELEASE_vX.Y.Z.md` 之类的独立文件
4. 发布完成后，下一次发版继续在同一个文件上覆盖更新

这样做的目的是降低维护成本，避免工作流配置、文档命名和实际发布内容不一致。

## 🚀 发布步骤

### 1. 更新版本号

版本号格式：`主版本号.次版本号.修订号+构建号`（例如：`1.1.0+1`）

#### 1.1 编辑 `pubspec.yaml`

打开 `pubspec.yaml` 文件，找到 `version` 字段：

```yaml
version: 1.1.0+1
```

根据发布类型更新版本号：
- **主版本号**（1.x.x）：重大更新、不兼容的 API 变更
- **次版本号**（x.1.x）：新功能、向后兼容的功能性更新
- **修订号**（x.x.0）：Bug 修复、小的改进
- **构建号**（+1）：每次构建递增，用于区分同一版本的不同构建

**示例：**
- 修复 Bug：`1.1.0+1` → `1.1.1+1`
- 新功能：`1.1.0+1` → `1.2.0+1`
- 重大更新：`1.1.0+1` → `2.0.0+1`

#### 1.2 提交版本号变更

```bash
git add pubspec.yaml docs/RELEASE_NOTES.md
git commit -m "chore: prepare release X.X.X"
```

#### 1.3 更新 `docs/RELEASE_NOTES.md`

在打 tag 前，先把 `docs/RELEASE_NOTES.md` 改成当前版本的发布说明，建议至少包含：

- 版本号
- 发布日期
- 新增内容
- 修复内容
- 兼容性说明或已知问题（如有）

### 2. 创建 Git Tag

#### 2.1 创建带注释的 Tag（推荐）

```bash
# 格式：v主版本号.次版本号.修订号
git tag -a v1.1.1 -m "Release version 1.1.1"

# 查看创建的 tag
git tag -l
```

#### 2.2 推送 Tag 到远程仓库

```bash
# 推送单个 tag
git push origin v1.1.1

# 或者推送所有 tag
git push origin --tags
```

### 3. 构建发布包

#### 3.1 构建 Android APK

**构建单个 APK（包含所有架构）：**

```bash
flutter build apk --release
```

构建完成后，APK 文件位于：`build/app/outputs/flutter-apk/app-release.apk`

**构建分架构 APK（推荐，文件更小）：**

```bash
flutter build apk --release --split-per-abi
```

构建完成后，APK 文件位于：
- `build/app/outputs/flutter-apk/app-armeabi-v7a-release.apk`
- `build/app/outputs/flutter-apk/app-arm64-v8a-release.apk`
- `build/app/outputs/flutter-apk/app-x86_64-release.apk`

#### 3.2 构建 Android App Bundle（可选，用于 Google Play）

```bash
flutter build appbundle --release
```

构建完成后，AAB 文件位于：`build/app/outputs/bundle/release/app-release.aab`

### 4. 创建 GitHub Release

#### 4.1 通过 GitHub Web 界面创建

1. 访问 GitHub 仓库页面
2. 点击右侧 "Releases" → "Draft a new release"
3. 选择刚才创建的 tag（例如：`v1.1.1`）
4. 填写 Release 标题（例如：`v1.1.1` 或 `Release 1.1.1`）
5. 在描述中填写更新内容（可以参考 CHANGELOG.md）
6. 上传构建好的 APK 文件
7. 勾选 "Set as the latest release"（如果是主要版本）
8. 点击 "Publish release"

#### 4.2 通过 GitHub CLI 创建（可选）

如果安装了 GitHub CLI，可以使用命令行创建：

```bash
# 安装 GitHub CLI（如果未安装）
# Windows: winget install GitHub.cli
# macOS: brew install gh
# Linux: 参考 https://cli.github.com/manual/installation

# 登录 GitHub CLI
gh auth login

# 创建 Release
gh release create v1.1.1 \
  --title "v1.1.1" \
  --notes "Release notes here" \
  build/app/outputs/flutter-apk/app-release.apk
```

### 5. 验证发布

发布完成后，请验证：

1. ✅ GitHub Releases 页面显示新版本
2. ✅ APK 文件可以正常下载
3. ✅ README.md 中的下载链接指向最新版本（如适用）

## 📝 完整发布命令示例

以下是一个完整的发布流程命令序列（以版本 `1.1.1` 为例）：

```bash
# 1. 确保代码已提交
git status

# 2. 更新版本号与发布说明（手动编辑 pubspec.yaml 和 docs/RELEASE_NOTES.md 后）
git add pubspec.yaml docs/RELEASE_NOTES.md
git commit -m "chore: prepare release 1.1.1"

# 3. 推送到远程
git push origin main

# 4. 创建并推送 tag
git tag -a v1.1.1 -m "Release version 1.1.1"
git push origin v1.1.1

# 5. 构建 APK
flutter build apk --release --split-per-abi

# 6. 创建 GitHub Release（通过 Web 界面或 CLI）
# 上传 build/app/outputs/flutter-apk/app-*-release.apk 文件
```

## 🔧 常见问题

### Q: 如何回退版本号？

如果版本号更新错误，可以：

```bash
# 1. 修改 pubspec.yaml 中的版本号
# 2. 提交更改
git add pubspec.yaml
git commit -m "chore: revert version to X.X.X+X"

# 3. 如果 tag 已推送，需要删除远程 tag
git tag -d v1.1.1          # 删除本地 tag
git push origin :refs/tags/v1.1.1  # 删除远程 tag
```

### Q: 如何查看所有已发布的版本？

```bash
# 查看所有 tag
git tag -l

# 查看 tag 详细信息
git show v1.1.1
```

### Q: 构建失败怎么办？

1. 清理构建缓存：
   ```bash
   flutter clean
   flutter pub get
   ```

2. 检查 Flutter 环境：
   ```bash
   flutter doctor
   ```

3. 重新运行代码生成：
   ```bash
   dart run build_runner build --delete-conflicting-outputs
   ```

## ⚠️ 注意事项

1. **版本号规范**：遵循语义化版本控制（Semantic Versioning）
2. **Tag 命名**：建议使用 `v` 前缀（如 `v1.1.1`）
3. **构建前检查**：确保代码生成已完成（`dart run build_runner build`）
4. **测试**：发布前务必在真实设备上测试 APK
5. **备份**：重要版本发布前建议创建分支备份
