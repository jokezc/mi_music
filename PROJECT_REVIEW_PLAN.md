# Project Review Plan

## Goal

分多轮对整个项目做主流程检查，覆盖主要用户入口、核心业务链路、状态同步、异常处理和代码封装质量。

原则：

- 不要求一次检查完全部内容
- 每一轮都要有明确范围和可交付结论
- 优先检查真实主流程和高风险状态链路
- 检查功能是否可用，也检查封装是否合理
- 每轮结束补充问题、修复项和后续建议

## Review Rounds

### Round 0: Baseline

目标：

- 梳理路由结构
- 梳理核心 Provider 依赖
- 梳理缓存、歌单、播放器三条核心数据流
- 为每个主要入口定义预期行为和风险点

输出：

- 入口清单
- 核心状态流说明
- 高风险区域列表

### Round 1: Login And Bootstrap

检查范围：

- 登录页
- 连接配置页
- 启动后的重定向逻辑
- 鉴权状态变化后的跳转
- 未登录、未配置、鉴权失效等异常态

重点：

- `router redirect` 是否稳定
- 启动时是否会出现错误跳转
- 认证失败后是否能正确回到登录态
- 登录成功后是否能恢复正常入口

输出：

- 登录链路问题清单
- 重定向逻辑建议
- 配置与认证状态收口建议

### Round 2: Library Main Flow

检查范围：

- Library 首页
- 歌单列表
- 管理歌单页
- 搜索入口
- 歌单详情页
- 收藏 / 系统 / 自定义歌单的行为差异

重点：

- 各类歌单的操作权限是否一致
- 刷新后 UI 和缓存是否一致
- 删除、移除、收藏等动作是否会引发状态错乱
- 页面之间是否复用了统一业务规则

输出：

- 音乐库主流程问题清单
- 歌单相关规则统一建议
- 缓存与展示同步建议

### Round 3: Player Main Flow

检查范围：

- 全屏播放页
- 播放队列
- 切歌
- 播放列表同步
- 删除歌曲 / 从歌单移除歌曲后的播放状态

重点：

- 当前歌曲、当前索引、当前歌单是否始终一致
- 刷新播放列表时是否存在边界错误
- 删除当前播放歌曲时行为是否稳定
- 播放页和队列页是否共享同一业务规则

输出：

- 播放链路问题清单
- 状态一致性建议
- 播放控制器封装建议

### Round 4: Download And Functions

检查范围：

- 下载页
- 功能页
- 连接页
- 定时任务列表页
- 定时任务编辑页

重点：

- 页面跳转是否顺畅
- 表单编辑与保存是否可靠
- 编辑后列表刷新是否正确
- 空态、异常态和失败提示是否完整

输出：

- 功能页链路问题清单
- 表单与列表状态建议

### Round 5: Settings Flow

检查范围：

- 设置首页
- 外观设置
- 客户端设置
- 关于页
- 设置项持久化和跨页生效

重点：

- 设置项修改后是否即时生效
- 持久化是否正确
- 重进页面后状态是否一致
- 设置页职责划分是否清晰

输出：

- 设置链路问题清单
- 设置状态管理建议

### Round 6: Cross-Cutting Code Review

检查范围：

- 公共工具类
- Provider 设计
- 页面职责划分
- 状态边界
- 重复业务逻辑

重点：

- 同一规则是否在多个页面各写一套
- 页面是否承担过多业务逻辑
- Provider 是否存在职责混杂或耦合过重
- 缓存刷新、播放刷新、UI 刷新是否有统一收口点

输出：

- 结构性问题清单
- 封装优化建议
- 模块边界建议

### Round 7: Final Summary

目标：

- 汇总所有轮次结果
- 归纳已修复问题
- 标记剩余风险
- 给出后续优化和自动化测试建议

输出：

- 总体结论
- 优先级排序后的整改建议
- 建议补充的测试清单

## Execution Notes

- 每一轮先做入口级检查，再追具体代码实现
- 每发现问题都要区分：
  - 功能缺陷
  - 状态一致性问题
  - 封装/结构问题
- 修复时优先统一规则来源，避免局部打补丁
- 每轮结束更新本文档，追加：
  - 已检查范围
  - 发现的问题
  - 已修复项
  - 待后续确认项

## Current Priority

建议先执行：

1. Round 0
2. Round 1
3. Round 2
4. Round 3

原因：

- 这是用户最常走的主流程
- 也是状态问题最容易聚集的区域
- 能尽早发现需要统一抽象的核心业务规则
